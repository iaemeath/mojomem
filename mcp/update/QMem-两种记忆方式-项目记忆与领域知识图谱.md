# QMem 两种记忆方式总结：项目记忆 vs 领域知识图谱

> **日期**：2026-07-17
> **定位**：本文是 QMem **架构蓝图的概念总纲**——把记忆切成"纵向项目动态域"和"横向共识域"两块，讲清各自本质、AI 如何理解、两者如何协同。
> **关联文档**：系统体检见 `QMem-FullAnalysis-2026-07-17.md`，机制细节见 `QMem-V3.2-Architecture.md`，V3.3 治理见 `QMem-V3.3-Architecture.md`，server-memory 参考实现对比见同目录后续文档。

---

## 一、总览对比

| 维度 | 项目记忆（MCP 模块） | 领域知识图谱 |
|---|---|---|
| **回答的问题** | "我们在哪？想怎么做？" | "我们在谈什么？底层逻辑是什么？" |
| **方向** | **纵向**（随项目演进） | **横向**（跨项目沉淀） |
| **性质** | 动态、私密、随时间流变 | 静态、专业、硬性约束 |
| **载体（QMem 落地）** | `tier=q4`（+ CBM 代码图） | `tier=consensus`（共识域） |
| **变化频率** | 高（每次编码都在改） | 低（沉淀后基本不动） |
| **谁写** | AI 编码时实时记录 | 用户/专家确认后沉淀 |

**核心判断**：这两种不是"二选一"，而是**分层治理、缺一不可**。项目记忆保证"工程落地连贯性"，领域知识图谱保证"业务思考专业度"。

## 二、项目记忆（MCP 模块）— 纵向动态域

### 2.1 定义

围绕**单个项目**的记忆，记录这个项目的架构、进度、踩坑、个人开发偏好。它是这个项目的"状态看板 + 工程记忆"。

### 2.2 内容构成（三层）

| 层 | 内容 | QMem 归属 |
|---|---|---|
| **客观状态** | 项目架构（前后端栈/端口/数据源）、任务进度（未开始→开发中→阻塞→完成）、已完成的功能 | `tier=q4`，type=progress / reference |
| **踩坑根因** | 这个项目特有的 bug 根因、字段陷阱、配置坑 | `tier=q4`，type=bugfix |
| **个人开发偏好** | 代码风格、异常处理习惯、注释风格、库选择倾向 | `tier=consensus` 的 `dev-preference` 域（跨项目稳定的个人风格） |

> **说明**：个人开发偏好虽然属于"项目记忆"范畴（它是"我想怎么做"），但因为**跨项目稳定**（同样的代码风格适用所有项目），所以归在 consensus tier 的一个独立域 `dev-preference`，而不是塞进每个项目的 q4。这是"性质归位"——稳定的进共识，易变的进 q4。

### 2.3 AI 的理解方式：状态机 + 看板

AI 必须精准知道：
- 当前系统拓扑（前端 Vue3 / 后端 SpringBoot 微服务 / 达梦库 / 端口分配）
- 每个任务的生命周期位置（哪个模块开发中、哪个阻塞、哪个完成）
- 用户希望代码长什么样（风格偏好）

### 2.4 检索特征

- **按项目隔离**：`mem_recall(current_project="changzhou-balance-plan")` 只查本项目记忆，不串味
- **高频读写**：每次编码都读（开场 `mem_context`）都写（收尾 `mem_save`）
- **易过期**：`progress` 类记忆超 30 天需复审（`review_queue` 拉模型提示）
- **配套 CBM**：代码实体（PO/DAO/Service/表字段/调用关系）不在 QMem 存，由 CBM 代码图谱承载（`search_graph` / `trace_path`）

## 三、领域知识图谱 — 横向共识域

### 3.1 定义

跨项目的**行业知识底座**，固化业务本体论、概念模型、因果关系。它是 AI 的"行业大脑"，防止 AI 用互联网大众语义去瞎猜专业概念。

### 3.2 内容构成（三类，对应共识域分治）

| 类 | 内容 | 共识域 | 例子 |
|---|---|---|---|
| **A. 技术陷阱共识** | 跨项目复现的硬技术坑 | `java-cloud-common` / `dameng-common` / `weakpwd` | IS_DELETE 中文值、CLOB 列长度、DM6/DM7 驱动差异 |
| **B. 业务领域知识** | 行业核心概念、因果约束、术语辨析 | `power-grid-domain` | 发电计划是核心概念不是属性、台账是模板实例不是 CRUD、负荷电量、D5000 表结构 |
| **C. 需求设计辨析** | 跨业务的方法论（模板vs实例、概念vs属性） | `requirement-design` | 数据建模时必须识别"类→实例"派生关系 |

### 3.3 AI 的理解方式：行业大脑 + 纠偏字典

没有领域知识图谱，AI 会犯三类降格错误：
- 把**核心业务概念**当成普通数据属性（"发电计划"→一张表的业务前缀）
- 把**派生关系**当成并列实体（"台账模板"和"台账数据"→两个独立 CRUD）
- 用**通识语义**瞎猜专业术语（"平衡"→会计平衡而非电网供需平衡）

有了它，AI 才明白业务概念之间的物理约束和因果律。领域知识图谱的**核心价值不是百科全书，而是"纠偏字典"**——只收 AI 容易理解错的概念。

### 3.4 检索特征

- **不绑项目**：`consensus_recall(query="发电计划")` 跨所有项目查（需求分析阶段尤其重要，此时还没进具体项目）
- **低频写入**：沉淀后基本不动，只在发现新概念/新陷阱时增补
- **稳定长效**：reference / decision 类，代码大改时才更新
- **图状关系**：概念之间有上下游/同层/约束关系。QMem 用 **content 内嵌的"关联概念"段落**表达（伪图方案），真正的图查询（多跳/路径/反向）由 CBM 承载

> **图结构分工**：领域知识图谱是"图状语义"，但**不做完整图数据库**。代码层面的实体/属性/关系/多跳查询交给 CBM（已有 Cypher 级查询 + Leiden 社区检测）；业务层面的概念关联写在 QMem content 文本里，靠 RAG 召回 + AI 读文本建立关联。两者不重叠。详见参考实现对比文档（server-memory 的弱图证明了"图数据结构≠图查询引擎"）。

## 四、两者如何协同（核心化学反应）

### 4.1 协同架构

```
┌──────────────────────────────────────────────────────────┐
│  AI 编码时，思考链并行调用两个域                          │
└────────────┬────────────────────────────┬────────────────┘
             ▼                            ▼
   ┌──────────────────┐         ┌────────────────────────┐
   │ 项目记忆域        │         │ 领域知识图谱域         │
   │ (QMem q4 + CBM)  │         │ (QMem consensus)       │
   │                  │         │                        │
   │ · 架构:微服务    │         │ · 业务:台账=模板实例   │
   │ · 进度:台账导入  │         │ · 因果:导入前必须校验  │
   │ · 偏好:async     │         │   模板字段定义         │
   │ · 代码实体(CBM)  │         │ · 纠偏:别建成两个CRUD  │
   └────────┬─────────┘         └──────────┬─────────────┘
            └──────────┬───────────────────┘
                       ▼
            符合个人风格 + 业务严谨的代码
```

### 4.2 真实场景演示

用户说：**"把台账导入模块的接口写一下，按我习惯来。"**

AI 内部并行检索：

1. **查项目记忆域**：
   - 进度 → 台账导入是 1.1 模块新任务
   - 架构 → SpringBoot 微服务 + 达梦 DM7 + cloud-frame
   - 偏好 → 用户要 `@Column(columnDefinition="CLOB")`、PO 要显式 schema、Controller 用 R 返回
   - CBM → `LedgerTemplatePO` 的字段、被谁调用

2. **查领域知识图谱域**：
   - 业务 → 台账数据是模板的**实例**，不是独立 CRUD
   - 因果 → 导入数据前必须先查模板元数据校验字段类型，否则数据污染
   - 纠偏 → 别把模板和数据建成两张无关联平表

3. **输出**：
   一段符合 cloud-frame 规范（项目记忆）、带 CLOB/schema 注解（个人偏好）、且内置"先查模板再校验实例"业务逻辑（知识图谱）的代码。

### 4.3 三者缺一的后果

| 缺什么 | 后果 |
|---|---|
| 缺项目记忆 | 代码不符合本项目架构（端口/数据源/返回格式全错），要返工 |
| 缺个人偏好 | 代码能跑但不像用户写的（风格不一致，每次都要手动改） |
| 缺领域知识图谱 | 代码能跑但业务逻辑错（漏掉模板校验，数据会脏；把核心概念降格为属性） |

## 五、本质区别（一句话）

| | 项目记忆 | 领域知识图谱 |
|---|---|---|
| **本质** | **工程记忆**（怎么造这个东西） | **世界知识**（这个东西本身是什么） |
| **类比** | 这栋楼的施工图纸 + 进度表 | 建筑学原理 + 材料力学 |
| **失效后果** | AI 不知道项目现状，重复造轮子 | AI 不懂业务，写出能跑但外行的代码 |

**项目记忆保证"工程落地连贯性"，领域知识图谱保证"业务思考专业度"。两者分层治理，缺一不可——这正是工业级 AI Agent 迈向实用化的终极形态。**

## 六、QMem 落地映射（实操速查）

把上面的概念映射到 QMem 实际工具和 tier：

| 概念 | QMem 落地 | 操作工具 |
|---|---|---|
| 项目架构/进度 | `tier=q4`，project=项目名 | `mem_save` / `mem_context` / `mem_recall(current_project=)` |
| 项目踩坑 | `tier=q4`，type=bugfix | `mem_save` |
| 个人开发偏好 | `tier=consensus`，域=`dev-preference` | `memory_promote(consensus_domain="dev-preference")` |
| 技术陷阱共识 | `tier=consensus`，域=`java-cloud-common` 等 | `memory_promote` + `consensus_recall` |
| 业务领域知识 | `tier=consensus`，域=`power-grid-domain` | `consensus_recall`（需求分析阶段主用） |
| 需求设计辨析 | `tier=consensus`，域=`requirement-design` | `consensus_recall` |
| 代码实体/调用图 | CBM 代码图谱（不进 QMem） | `search_graph` / `trace_path` / `get_architecture` |

### 写入纪律（避免两类记忆串味）

1. **易变的进 q4**：进度、当前架构状态、未推送的改动 → `tier=q4` + type=progress
2. **稳定的进 consensus**：跨项目验证过的陷阱、业务概念、个人偏好 → `tier=consensus`
3. **代码事实不进 QMem**：函数签名/调用关系/表字段 → 查 CBM，QMem 只存"代码里读不出的决策/背景/概念"
4. **需求分析阶段只走 consensus**：`mem_recall` 会混入 q4 草稿污染概念检索，需求分析/语义理解阶段统一用 `consensus_recall`
