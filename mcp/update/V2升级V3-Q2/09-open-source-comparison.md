# 开源工具横向对比：如何处理"共识"

> 状态：参考资料
> 时间：2026-07-13
> 背景：在双库方案和三表方案之间决策时，调研开源记忆工具的做法

## 核心发现

> **开源记忆工具没有一个把记忆按"共识/非共识"物理分库。**

Mem0、Letta、Zep、Cognee——全都用统一存储 + metadata/标签区分。所谓"共识"只是"被多次验证、权重高的记忆"，不是物理分库。

## 各工具做法

### Mem0

| 维度 | 做法 |
|---|---|
| 存储 | facts + embedding + graph，同库 |
| 区分类型 | `memory.add(data, metadata={type:'consensus'})` |
| 检索 | `search(query)` 统一向量检索，可选 metadata 过滤 |
| 共识处理 | 不分库，靠 metadata 标签 |

**关键设计**：LLM 自动决定提取/更新/删除哪些记忆。记忆生命周期由 AI 管理，不靠人工 promote。

### Letta（MemGPT）

| 维度 | 做法 |
|---|---|
| 存储 | 分层记忆（core memory / archival memory），但都在同一个系统内 |
| 区分类型 | 不区分。archival memory 统一存，按相关度检索 |
| 检索 | LLM 通过工具调用 `archival_memory_search`，自己做检索决策 |
| 共识处理 | 不区分类型。所有记忆统一存，按相关度排序 |

**关键设计**：LLM 自己管理记忆的分页和检索，类似操作系统的内存管理。

### Zep

| 维度 | 做法 |
|---|---|
| 存储 | 时序知识图谱（节点 + 边） |
| 区分类型 | 节点带类型标签 |
| 检索 | 图查询 + 向量搜索 |
| 共识处理 | 不分库。节点类型标签区分 |

### LangChain RouterChain

| 维度 | 做法 |
|---|---|
| 路由方式 | LLM/embedding 做路由决策 |
| 目标 | 都是同一记忆层，不是分库 |

## 为什么开源工具不分库

### 原因 1："是否共识"是会变的属性

今天某条经验是单项目的 bug，明天发现 3 个项目都踩了，就变成共识了。

- **分库方案**：每次"升级为共识"都要搬数据（Q4→Q2），"降级"也要搬回来
- **标签方案**：只需改一个字段（metadata / tier / type）

### 原因 2：检索不应该割裂

用户问"这个框架有什么坑"，答案可能既有动态记忆（项目特有的坑）也有共识（通用坑）。分库后需要跨库联合检索，增加复杂度。

### 原因 3：物理分库的收益在数据量大时才明显

Mem0 的百万级 memory 才需要分 collection。几十到几百条记忆的规模，单库完全够用。

## 对 QMem 设计的启发

### 启发 1：标记而非搬运

> Mem0 用 metadata JSON，QMem 的单表 tier 方案用独立列 tier，思路相同——标记而非搬数据。

这验证了"标记优于搬运"的方向。但 QMem 的场景有个 Mem0 没有的需求：**项目与共识的引用关系**。Mem0 的记忆是平的，没有"项目 A 引用共识 B"的概念。QMem 需要这个，所以单表 tier 不够，需要加关联表。

### 启发 2：AI 自己做检索决策

> Letta 让 LLM 通过工具调用 `archival_memory_search`，自己做检索决策。

QMem 的 `consensus_recall` 工具就是这个模式——AI 判断"这个问题该搜共识还是搜动态记忆"。但这带来工具选择负担。Letta 的解决方案是让 AI 自然地做这个决策（通过系统提示训练），而不是靠物理分库强制。

### 启发 3：统一检索 + 过滤 > 分库检索

> 所有开源工具都是统一检索 + 条件过滤，不是分库后分别检索。

QMem 的三表方案虽然物理分表，但 `mem_recall` 在应用层做了联合检索（搜 project_memory + consensus_memory 合并结果），实际上模拟了"统一检索 + 过滤"。这是正确的方向——物理分表是为了隔离和数据完整性，不是为了割裂检索。

## 最终对 QMem 的指导

| 开源工具原则 | QMem 三表方案的对应 |
|---|---|
| 标记而非搬运 | promote 搬行但 embedding 不重算（搬 bytes） |
| 统一检索 + 过滤 | mem_recall 应用层联合两表检索 |
| 引用关系显式化 | project_consensus 关联表（开源工具没有，QMem 特有需求） |
| AI 做检索决策 | consensus_recall 独立工具，AI 按需调用 |

**三表方案本质是"开源工具的标记思路 + QMem 特有的引用关系需求"的结合。** 物理分表是为了让引用关系有清晰的边界（哪些是动态记忆、哪些是共识），不是为了割裂检索。
