# 方案 V2.0：is_global 标记 + RRF boost

> 状态：❌ 被否决（隔离不彻底，boost 权重不可控）
> 时间：2026-07-09（QMem 初版，从 Engram 迁移）

## 核心设计

共识记忆和动态记忆存在**同一张表** `memory_facts`，靠 `is_global` 字段（0/1）区分。promote 操作只是改一个标记：

```sql
-- schema
is_global INTEGER NOT NULL DEFAULT 0

-- promote
UPDATE memory_facts SET is_global=1 WHERE obs_uuid=?
```

检索时，`search_rrf.py` 的 RRF 融合对 `is_global=1` 的条目做三重加权：

```python
# 1. 词法路命中：×1.5
boost = 1.5 if item.get("is_global") else 1.0
scores[iid] += boost * (1.0 / (k + rank + 1))

# 2. 语义路命中：×1.5
boost = 1.5 if item.get("is_global") else 1.0
scores[iid] += boost * (1.0 / (k + rank + 1))

# 3. 固定加分（等效 rank=0 垫底）
GLOBAL_BONUS = 1.0 / (k + 1)
if item.get("is_global"):
    scores[iid] += GLOBAL_BONUS
```

## 优点

- **promote 极简**：一行 UPDATE，不搬数据，embedding/FTS 全保留
- **检索完整**：RRF 两路都覆盖，共识和动态记忆一起排序返回
- **可逆**：降级就是改回 `is_global=0`
- **实现成本低**：加一列 + boost 逻辑

## 缺点

### 1. 隔离不彻底
共识和日常记忆在**同一张表**。靠 boost 区分，但 boost 权重是经验值（1.5/GLOBAL_BONUS），共识条目多到几十条时，boost 效果被稀释，和噪声记忆重新混在一起。

### 2. boost 权重不可控
`×1.5` 是拍脑袋的。太低 → 共识排不上去；太高 → 共识淹没动态记忆。没有自适应机制，条目数变化时权重需要手动调整。

### 3. is_global 语义模糊
"全局"是什么意思？全局共识？哪个层级的全局？字段名不清晰，容易误解。

### 4. 无法显式表达"谁引用了谁"
"bfo_zj_yxyd 和 _java-cloud-common 的关系"是隐式的。没有结构化的引用关系，只能靠 project 名前缀（`_` 开头）猜测。

## 为什么被否决

核心问题：**共存但加权**是一个脆弱的妥协。共识和动态记忆混在一起，靠权重区分本质上是概率性的——有时能区分，有时不能。随着共识数量增长，区分能力持续下降。
