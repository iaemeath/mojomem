# QMem 架构拷问深度解析 (FAQ)

本文档整理了系统在架构设计与理论升维过程中，面临的三个最核心的拷问与解答。

## Q1: Q2 (第二象限) 既然全是 `.md` 且不进向量库，AI 如何拉取它？
原版《PRQ-Four Quadrants》方案中，Q2（全局公共资产）被要求绝对隔离，存放在 `D:\code\.assets\*.md`。如果脱离了向量库的模糊检索，AI 怎么知道去拉取特定的 `.md` 文件？

**解答与原理解析：**
这套方案极度依赖 **Q3（第三象限户口本）的指针机制**。
1. **冷启动埋点**：当我们在项目根目录执行 `/init` 时，生成的 `CLAUDE.md` (Q3) 底部会被强行写入一句显式的引用指针，如 `Ref: D:\code\.assets\vue2-common.md`。
2. **Push 注入大脑**：由于 `CLAUDE.md` 在启动时会被 AI 客户端通过系统级 Prompt 前置读取（Push），AI 潜意识中已经知晓了这个物理指针的存在。
3. **显式拉取**：在后续遇到前端报错时，AI 能够根据记忆中的指针，通过 `mem_recall` 精确传入该文件名，从而实现按需拉取。

*(注：当前 Python 版使用 `is_global=1` 将 Q2 提权混入 Q4 向量库，是一种技术上更讨巧的妥协方案，它利用 RRF 解决了“模糊匹配”的需求，但牺牲了绝对的物理隔离。)*

## Q2: 为什么 `mem_delete` 显得不那么重要了？(Semantic Overwrite)
在早期的 RAG 系统中，记忆库总是随着时间变成“垃圾场”，因此不得不开发各种复杂的遗忘机制和清理脚本。

**解答与原理解析：**
QMem (Python 版) 在 `mem_save` 底层实现了基于 `topic_key` 的 **语义覆盖 (Semantic Overwrite)**。
- 这是一种“进化式”的记忆修剪。如果旧的网关 IP 变了，或者某个旧的踩坑记录有了更优解，大模型只需要使用相同的 `topic_key` 写入新规则。
- 底层接口会自动执行 `Upsert`（更新文本，并重算覆盖旧向量），无需显式调用 `mem_delete`。
因此，`mem_delete` 只有在某个规约“彻底作废且没有新内容替代”时才需要被人工或 AI 触发。

## Q3: 为什么 QMem 坚决抵制企业招聘中火热的 LangGraph、Milvus 等技术？
外面企业对 Agent 开发岗位的招聘要求，动辄掌握 LangGraph、CrewAI 多智能体框架，以及 Pinecone、Milvus 等重量级向量数据库。为什么 QMem 完全不用？

**解答与原理解析：**
因为 QMem 的设计哲学是**“Unix 极客式的克制”**与**“单兵作战辅助”**，这与上述重型技术的应用场景完全对立。

1. **抵制 LangGraph / CrewAI (编排框架)**：
   - 那些框架用于云端写多 Agent “踢皮球”式的复杂流转脚本。
   - QMem 定位是 **底层的极速存储引擎 (Memory Engine)**。我们通过标准 MCP 协议暴露出“推、拉、查”四个原子接口，不负责教 AI 怎么思考。引入重型编排框架会让系统臃肿不堪，偏离初衷。
2. **抵制 Pinecone / Milvus (重量级向量库)**：
   - 它们是为云端亿级数据设计的，必须运行在 K8s 或 Docker 容器中。
   - QMem 面对的是内网 2000 个仓库的个人开发机环境。我们必须保证**离线、零运维、秒级起停**。使用 `SQLite3 + sqlite-vec` 扩展，将整个向量和文本库缩减为一个几 MB 的 `.db` 单文件，才是工业级落地的神来之笔。
3. **吸收了什么？(高级 RAG)**：
   - 我们抛弃了沉重的 LlamaIndex，但原汁原味地提取了它最核心的思想：**双路混合检索与 RRF（倒数排名融合）重排**。不到 200 行 Python 代码，就解决了 RAG 容易“胡说八道”和“断章取义”的顽疾。
