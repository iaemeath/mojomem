# 🚀 工业级 AI 编程长记忆系统（PRQ-Four Quadrants）开发指南

本指南旨在为全内网、全离线、多项目群并发（2000+ 仓库）的 Windows 11 开发环境，提供一套去中心化、单库闭环、绝对掌控的 AI 长记忆系统（系统代号：QMem / mojomem）的重构与开发标准。

---

## 🏛️ 一、 方案核心思想：PRQ 三模驱动

本方案彻底抛弃传统的“聊天记录盲目 RAG”机制，回归 Unix 的减法哲学与文件系统范式。AI 记忆的加载与吞吐严格遵循 **PRQ 三模触发机制**，从底层逻辑上精确控制 Token 消耗与上下文的绝对理智：

1. **Push（推 — 被动注入机制）**：
   * **本质**：启动会话或切入目录时，由系统硬性、前置地将最高规范作为 System Prompt 强行“压入”大模型大脑，AI 动笔前必须无条件服从。
2. **Pull（拉 — 动态召回机制）**：
   * **本质**：在编码或 Debug 过程中，大模型遇到未知报错或需要回溯历史时，主动发出特定指令，从本地长记忆账本中“捞取”相关的踩坑痕迹。
3. **Query（查 — 刚性结构检索）**：
   * **本质**：AI 需要评估代码修改的“爆炸半径”时，主动通过 AST（抽象语法树）分析工具查询物理代码事实，不涉及任何主观聊天记忆。

## 🗺️ 二、 升级版：四象限记忆拓扑模型

记忆体系依照“变动频率（静态 vs 动态）”与“作用范围（全局公共 vs 项目私有）”两个核心维度，划分为以下四大物理载体。**严禁各象限在存储介质上发生交叉混淆**：

```text
                        【 作用范围：全局公共 (Global) 】
                                         │
        第一象限：【全局环境红线·静态】         │   第二象限：【项目群公共资产·动态】
        (内网私服、总网关、全局安全硬指标)     │   (跨项目可复用的高阶解决方案、避坑Pattern)
        ───────────────────────────────  │   ────────────────────────────────────────
        载体：D:\code\CLAUDE.md          │   载体：D:\code\.assets\*.md 模块化文件
        模式：Push (被动强制注入)          │   模式：Semi-Push / 显式 Pull 召回
                                         │
─────────────────────────────────────────┼─────────────────────────────────────────►【 变动频率 】
  (几乎不改，启动即常驻)                   │                  (随开发动态增量写入)
                                         │
        第三象限：【项目身份与硬约束·静态】     │   第四象限：【项目动态踩坑本·动态】
        (绝对地域声明、Vue2/3死守、特定网关)   │   (Git分支足迹、临时会话交接、当下Bug)
        ───────────────────────────────────│   ─────────────────────────────────────
        载体：各子项目/CLAUDE.md          │   载体：单库 SQLite (core_memory.db)
        模式：Push (路径穿透激活)          │   模式：Pull (物理级双路混合检索)
                                         │
                        【 作用范围：项目私有 (Local) 】
```

## 📂 四象限物理边界与存储规范

### 📌 第一象限：全局环境红线 (Global & Static)
* **存储载体**：总控制上级目录物理文件 `D:\code\CLAUDE.md`。
* **核心内容**：内网 NPM / Maven / Docker 私服镜像源绝对地址；全局大网关路由分发规则；公司级刚性安全拦截指标（如弱口令过滤函数命名规范）。
* **触发机制**：Push。由于在 `D:\code` 启动，AI 在第一秒将其作为全局背景常识加载。

### 📌 第二象限：项目群公共资产 (Global & Dynamic)
* **存储载体**：独立于数据库之外，存放于 `D:\code\.assets\` 目录下的模块化纯文本 Markdown 文件（如 `vue2-common.md`, `spring-cloud-infra.md`）。
* **核心内容**：特定技术栈经过长期验证的通用避坑 Pattern（如旧版 Element-UI 弹窗遮罩层 z-index 冲突的 CSS 刚性修补解法、特定内网旧依赖排除方案）。
* **隔离设计**：**绝对禁止将第二象限塞入第四象限的向量数据库中**。通过独立的 `.md` 抽屉式文件存放，彻底切断前端技术栈与后端 Java 逻辑在模糊向量空间中可能发生的“记忆串流”与上下文污染。

### 📌 第三象限：项目身份与硬约束 (Local & Static)
* **存储载体**：各个子项目 Git 仓库根目录下的 `各项目/CLAUDE.md`。
* **核心内容**：绝对的地域分支声明（如：“当前为泰州分支，严禁抄袭靖江分支配置”）；技术栈代差死命令（如：“本项目强制死守 Vue 2.6 Options API，严禁引入 Vue 3 语法”）；本地启动与编译指令。
* **开荒冷启动机制**：借力打力。在陌生项目克隆到本地后，直接在子目录运行 AI-Agent 原生的 `/init` 启动命令。通过注入全局规约，劫持其扫描行为，让大模型自动在几秒内吐出符合该项目特征的户口本 `CLAUDE.md`，并在文件末尾自动追加对第二象限的物理引用行（如 `Ref: ../.assets/vue2-common.md`）。

### 📌 第四象限：项目动态踩坑本 (Local & Dynamic)
* **存储载体**：单文件 SQLite 数据库（`core_memory.db`）。内置 `sqlite-vec` 向量虚表及级联触发器（Trigger）。
* **核心内容**：高频流动的战术错题本。包含近期的 Git 分支开发足迹、临时会话的交接单（Handoff Status）、当前环境特定组件的 Bug 修复日志。
* **清除机制**：完全摒弃 ADD-only 的脏数据堆积。旧决策一旦过时，由人类或 AI 发起删除命令，在数据库底层通过原子级事务与触发器一并硬抹除。

## 🛠️ 三、 Mojo 引擎（mojomem）底层开发规范

### 1. 物理环境与依赖约束
* **目标平台**：Windows 11 x64 (纯全内网离线开发机)。
* **运行期依赖**：拒绝常驻的 Python Watcher 轮询服务，拒绝笨重集群。轻量级挂载 ONNX Runtime (CPU 版)，离线加载 `bge-small-zh-v1.5`（95MB）。
* **存储依赖**：集成本地单文件 SQLite 3，引入 `sqlite-vec`。

### 2. 双路混合检索（Hybrid Search）算法实现
由于原生 FTS5 是一个中文分词残疾，Pull 召回工具必须在内部使用 **FTS5 精确硬标签（Topic Key）+ 传统 LIKE 字面模糊 + sqlite-vec 向量余弦距离** 的多路归并排序（RRF 算法）。

```text
[ 大模型发起 Pull 召回 ] ➔ "查询泰州项目关于弱口令校验的暗坑"
           │
           ├──► 语义路：ONNX 算得 BGE 向量 ──► sqlite-vec 计算 Cosine 相似度
           │
           └──► 字面路：SQL 原生 LIKE '%弱口令%' ──► 100% 刚性捞出特异性业务关键词
                       │
                       ▼
            [ 两路结果就地求并集并去重 ]
                       │
                       ▼
         Push 给大模型，单次查询延迟 < 5ms
```

### 3. 暴露给大模型的最小 MCP 接口规范
仅向外暴露 4 个干脆、高优先级的核心工具，拒绝多余功能：
* `mem_save(project, topic_key, content)` [Pull 写入]：原子事务将文本与 512 维向量一并拍入 `core_memory.db`。
* `mem_recall(query, current_project)` [Pull 召回]：执行双路混合检索。
* `memory_promote(obs_id)` [微操升级]：将某条私有踩坑记录一键提纯。执行 `UPDATE memory_facts SET project = 'GLOBAL' WHERE id = ?`，使其瞬间飞入主账本，辐射全局。
* `mem_delete(obs_id)` [硬擦除]：触发底层 Trigger 同步强力清洗。

## 🚀 四、 开发者冷启动工作流指南

1. **第一步：宪法打底（Push）**：在开发机根目录（如 `D:\code`）丢入第一象限 `CLAUDE.md`。关闭 AI 代码工具自带的 Auto Memory 干扰。
2. **第二步：冷启动陌生仓库（/init）**：克隆目标项目后，进入目录执行 `claude /init`。大模型自动盘点出该项目的**第三象限户口本 (`CLAUDE.md`)**，并连上第二象限 `.assets/` 文件的引用链。
3. **第三步：守护第四象限（Pull & Query）**：启动 QMem MCP 服务。大模型遇到复杂 Bug 时自动通过 `mem_save` 存入 SQLite 错题本。
4. **第四步：微操净化**：配置发生真实变更时，人肉或通过 AI 呼叫 `mem_delete` 对记忆执行物理擦除。
